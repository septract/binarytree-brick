# BRiCk Verification Plan for Daedalus Red-Black Tree

*Created: 2026-02-13*

## Motivation

CBMC verification of the Daedalus C++ red-black tree (`ddl/map.h`) failed
because CBMC's hand-maintained C++ parser crashes on
`std::numeric_limits<size_t>::max()` in `ddl/size.h` — even though that code
is dead from `map.h`'s perspective. See
[`cbmc/2026-02-12_cbmc_cpp_support.md`](../cbmc/2026-02-12_cbmc_cpp_support.md)
for the full investigation.

**BRiCk** ([bedrocksystems/BRiCk](https://github.com/bedrocksystems/BRiCk))
is a framework for formally verifying C++ code without rewriting it. It uses
Clang's fully-elaborated AST (so templates, `if constexpr`,
`std::numeric_limits` are all resolved before translation), then generates a
Coq deep embedding for separation logic proofs via the Iris framework.

## Local Toolchain Setup

All tools are installed locally via opam (no Docker).

### Prerequisites

- **opam** >= 2.1 (`brew install opam`)
- **Homebrew LLVM** (`brew install llvm`) — needed for cpp2v's libclang dependency
- **Apple Clang** is NOT sufficient; cpp2v links against LLVM's C++ libraries

### Installation

```bash
cd daedalus-rb/brick
make setup    # Creates opam switch 'brick' with Coq + Iris + cpp2v (~30 min)
```

This runs:
1. `opam switch create brick 5.1.1` — fresh OCaml 5.1.1 switch
2. Adds coq-released and iris-dev opam repositories
3. Pins `coq-cpp2v-bin` to the BRiCk GitHub repo
4. Installs `coq-cpp2v-bin` with all dependencies (Coq, Iris, cpp2v binary)

After setup, activate the switch:
```bash
eval $(opam env --switch=brick)
```

### Verification that setup works

```bash
cpp2v --version     # Should print cpp2v version
coqc --version      # Should print Rocq/Coq version
```

## Architecture

```
daedalus-rb/brick/
├── ddl/                  # Unmodified Daedalus headers (copied from cbmc/ddl/)
│   ├── map.h             # The C++ code under verification
│   ├── boxed.h           # Reference counting (HasRefs, Boxed<T>)
│   ├── size.h            # Size type (contains std::numeric_limits — OK for Clang)
│   ├── maybe.h           # Optional type
│   └── debug.h           # Debug macros (no-op at DEBUG_LEVEL=0)
├── src/
│   └── map_int_int.cpp   # Monomorphized driver for cpp2v
├── coq/
│   ├── _CoqProject       # Coq project file
│   ├── RBTree.v          # Functional spec (ported from Lean)
│   ├── TreeRep.v         # Separation logic representation predicate
│   ├── FindSpec.v        # FindNode spec + proof scaffold
│   ├── InsertSpec.v      # Insert spec + proof scaffold
│   ├── RefCount.v        # Reference counting correctness
│   └── Invariants.v      # End-to-end glue proofs
└── Makefile              # setup, cpp2v, proofs, clean targets
```

Generated by cpp2v (not checked in):
- `coq/map_int_int_cpp.v` — Deep embedding of the C++ AST
- `coq/map_int_int_cpp_names.v` — Symbol table for mangled C++ names

## Monomorphization Strategy

cpp2v operates on Clang's elaborated AST, so templates must be instantiated.
`src/map_int_int.cpp`:

1. `#include <ddl/map.h>` — pulls in all Daedalus headers
2. `template class DDL::Map<int, int>;` — explicit instantiation forces
   Clang to emit full AST nodes for all `Map<int,int>::Node` methods
3. A `driver()` function calls each operation to verify (insert, contains,
   lookup, valid, copy, free)

Unlike CBMC, BRiCk uses Clang natively — `std::numeric_limits`,
`std::is_base_of`, `<iostream>` all work. No stubs needed.

## Proof Strategy: Refinement Against Lean Spec

The existing Lean proofs (`Rbtree/Daedalus.lean`, `Rbtree/Daedalus/Proofs.lean`)
establish the functional spec is correct. We:

1. **Port the spec to Coq** in `RBTree.v`: `Color`, `tree`, `isBlack`, `isRed`,
   `setRebalanceLeft`, `setRebalanceRight`, `ins`, `makeBlack`, `insert`,
   `findNode`, `ForAll`, `IsBST`, `NoRedRed`

2. **Define separation logic `tree_rep`** in `TreeRep.v`: links the Coq `tree`
   type to the C++ `Node` heap layout using BRiCk's points-to assertions

3. **Prove each C++ function refines its spec** counterpart:
   - `FindSpec.v`: `findNode` (read-only, simplest proof)
   - `InsertSpec.v`: `ins`, `insert`, `setRebalanceLeft`, `setRebalanceRight`,
     `makeCopy`
   - `RefCount.v`: `copy`, `free` (Iris ghost state for ownership tokens)

4. **Compose with invariant proofs** in `Invariants.v` for end-to-end guarantees

## Risks and Mitigations

| Risk | Status | Mitigation |
|------|--------|------------|
| cpp2v incompatible with LLVM 21 (Homebrew) | **Resolved** | Works with LLVM 21.1.8 on Apple Silicon |
| opam package resolution conflicts | **Resolved** | Built cpp2v from source via cmake; Coq+Iris installed via opam separately |
| cpp2v chokes on `operator<<` / `<iostream>` | **Resolved** | cpp2v emits warnings for atomics/NEON but produces correct output |
| BRiCk Coq theories incompatible with generated AST | Open | Generated AST uses `skylabs.lang.cpp.parser`; need matching theory version |
| Insert proof too complex | Open | Start with `Admitted`; prove incrementally case-by-case |

## Verification Scope

**In scope** (core operations):
- `insert` / `ins` — functional correctness + invariant preservation
- `findNode` — correctness (simplest proof, good warm-up)
- `copy` / `free` / `makeCopy` — reference counting soundness

**Out of scope** (stretch goals):
- `Iterator` — complex stateful traversal
- `compare`, `operator==` — comparison operators
- `operator<<`, `dump`, `toJS` — I/O and serialization
